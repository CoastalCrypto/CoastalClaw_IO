## 1. Release Framing

Two coupled themes:

1. **Architect 2.0** (human-built) — turn the toy `coastal-architect` loop into a credible self-improvement system: work-item queue with three input adapters, multi-file diffs, ModelRouter integration, per-item approval policy, PR creation, staged review, observability dashboard, plain-English operator UX.
2. **Reliability dogfood** (architect-built) — the *current* architect runs against a seeded `skill_gaps.db` (real telemetry replay + curated tool-file weak points), proposes fixes on `packages/core/src/tools/core/{shell,file,sqlite,web,git}.ts`, the green ones get cherry-picked into v1.5.0 with `Co-Authored-By: coastal-architect <bot@coastal.ai>` attribution.

### Release narrative (CHANGELOG line)

> *v1.5.0 ships the upgraded Architect: it now takes work items, writes plans, runs against your real models, opens PRs, and shows its work. As proof, the previous architect contributed N reliability fixes to this very release — receipts in the changelog.*

### Out of scope for v1.5.0 (explicit YAGNI)

- Architect cannot write tests it does not already have a fixture for (single-file scope for tests stays).
- No autonomous mode for `full` approval-policy items — humans always gate creation of net-new files in `full` mode.
- No "architect chats with you to refine the plan" — plan stage is one-shot generate, approve / revise / reject only.
- No multi-architect concurrency — one cycle at a time, work items processed serially.
- No GitHub Actions integration for CI mirror of the build gates — local `pnpm` only for v1.5.0.
- No rollback automation in the architect's normal flow — reverts via Time-Travel snapshots are operator-initiated only (never auto-merge to main).
- No Curriculum mode that proposes work outside `packages/`, `coastalos/`, `scripts/` — it cannot touch docs, packaging, or release infra without an explicit human-created work item.
- No skill marketplace / discovery in v1.5 — SKILL.md export/import is shipped, discovery deferred to v1.6 (see §7).

## 2. Architecture & Data Model

### New SQLite store

`data/architect.db` — separate from `data/skill-gaps.db` so the legacy reactive system does not entangle with the new queue-driven system. The dogfood plan (Section 6) bridges the two by writing into `skill-gaps.db` for the *old* architect; the new architect reads `architect.db`.

### Tables

```sql
-- Unified work queue. All input adapters write here.
CREATE TABLE work_items (
  id              TEXT PRIMARY KEY,           -- ulid
  source          TEXT NOT NULL,              -- 'ui' | 'markdown' | 'skill_md' | 'github' | 'skill_gap' | 'curriculum'
  source_ref      TEXT,                       -- GH issue number, .architect/queue.md hash, gap_id
  title           TEXT NOT NULL,
  body            TEXT NOT NULL,
  target_hints    TEXT,                       -- JSON array of file globs/paths
  acceptance      TEXT,                       -- free-text criteria
  budget_loc      INTEGER NOT NULL DEFAULT 200,
  budget_iters    INTEGER NOT NULL DEFAULT 5,
  approval_policy TEXT NOT NULL DEFAULT 'plan-only',  -- full | plan-only | pr-only | none
  review_timeout_min INTEGER NOT NULL DEFAULT 60,
  on_timeout      TEXT NOT NULL DEFAULT 'reject',     -- revise | reject | auto_approve
  status          TEXT NOT NULL DEFAULT 'pending',  -- see Status Vocabulary below
  paused_reason   TEXT,                       -- NULL unless status='error' or 'paused'
  paused_at       INTEGER,                    -- timestamp of pause; cleared on resume
  resumable       INTEGER NOT NULL DEFAULT 0, -- bool: explicitly marked resumable by operator action
  recurrence_count INTEGER NOT NULL DEFAULT 0,-- how many times this item has hit the same hard-fail across resumes
  escalated_at    INTEGER,                    -- timestamp when recurrence_count crossed threshold (default 3)
  created_at      INTEGER NOT NULL,
  updated_at      INTEGER NOT NULL,
  CHECK (status IN ('pending','active','awaiting_human','merged','cancelled','error','paused'))
);

-- One row per architect attempt against a work item.
CREATE TABLE cycles (
  id              TEXT PRIMARY KEY,           -- ulid
  work_item_id    TEXT NOT NULL REFERENCES work_items(id),
  iteration       INTEGER NOT NULL DEFAULT 1,
  stage           TEXT NOT NULL,              -- see Status Vocabulary below
  plan_text       TEXT,
  diff_text       TEXT,
  branch_name     TEXT,
  pr_url          TEXT,
  test_summary    TEXT,
  model_used      TEXT,                       -- e.g. 'vllm:llama-3.1-70b'
  failure_kind    TEXT,                       -- NULL on success; otherwise: parse|apply|locked|budget|test|type|lint|build|env_*
  revise_context  TEXT,                       -- JSON: { from_cycle, comment, test_output, pr_comments }
  duration_ms     INTEGER,
  outcome         TEXT CHECK (outcome IN ('merged','failed','vetoed','error','revised') OR outcome IS NULL),
  error_message   TEXT,
  created_at      INTEGER NOT NULL,
  updated_at      INTEGER NOT NULL
);

-- Approval-gate audit trail.
CREATE TABLE approvals (
  id              TEXT PRIMARY KEY,
  cycle_id        TEXT NOT NULL REFERENCES cycles(id),
  gate            TEXT NOT NULL,              -- plan | diff | merge
  decision        TEXT NOT NULL,              -- approved | rejected | revised | timeout | auto
  decision_revise INTEGER NOT NULL DEFAULT 0, -- bool
  decided_by      TEXT,                       -- user id or 'system'
  comment         TEXT,
  created_at      INTEGER NOT NULL
);

-- Workspace snapshots (Agent-Zero-style Time Travel) — shadow-Git references.
CREATE TABLE snapshots (
  id              TEXT PRIMARY KEY,           -- ulid
  cycle_id        TEXT REFERENCES cycles(id), -- NULL for snapshots not tied to a cycle (e.g. manual)
  work_item_id    TEXT REFERENCES work_items(id),
  shadow_ref      TEXT NOT NULL,              -- sha in data/.architect-snapshots/.git
  parent_id       TEXT REFERENCES snapshots(id),
  captured_at     INTEGER NOT NULL,
  captured_by     TEXT NOT NULL,              -- 'auto:building' | 'auto:plan' | 'manual:<user_id>'
  note            TEXT,                       -- e.g. "before applying diff for cycle 01HXY..."
  retention       TEXT NOT NULL DEFAULT 'short' -- 'short' (30d) | 'long' (1y) | 'pinned' (forever)
);

-- Pre-aggregated metrics (rebuild triggers documented below — no scheduler) for instant Insights page render.
CREATE TABLE cycle_metrics (
  date           TEXT NOT NULL,                -- YYYY-MM-DD
  metric         TEXT NOT NULL,                -- success_rate | failure_kind:test | etc.
  value          REAL NOT NULL,
  count          INTEGER NOT NULL,
  PRIMARY KEY (date, metric)
);
```

### Status Vocabulary (authoritative)

Two distinct enums — one per entity. Implementations must not mix them.

**`work_items.source`** (where the work item came from — for attribution and Receipts):

| Value | Meaning |
|---|---|
| `ui` | Created via dashboard form |
| `markdown` | Imported from `.architect/queue.md` |
| `skill_md` | Imported from `.architect/skills/*.md` (Agent Zero / Space Agent SKILL.md format) |
| `github` | Pulled from a labeled GitHub issue |
| `skill_gap` | Created from `skill-gaps.db` via legacy bridge |
| `curriculum` | Self-proposed by the architect during a curriculum scan |

**`work_items.priority`** (scheduling order):

| Value | Meaning |
|---|---|
| `high` | Front of queue (used sparingly — emergency fixes, time-sensitive items) |
| `normal` | Default for ui / markdown / skill_md / github sources |
| `low` | Default for curriculum source. Yields to any normal/high item that arrives. |

**`work_items.status`** (lifecycle of a request):

| Value | Meaning | Set by |
|---|---|---|
| `pending` | Created, not yet picked up | adapter on insert |
| `active` | A cycle is running for this item (any stage) | runner on cycle start |
| `awaiting_human` | Cycle parked at a review gate (`plan_review` or `pr_review`) | runner on gate open |
| `merged` | A cycle reached `merged` outcome — terminal | runner on merge |
| `cancelled` | Rejected by human OR `budget_iters` exhausted — terminal | runner / human action |
| `error` | Hard fail at any stage — recoverable, no cycle running | runner on hard fail |
| `paused` | Operator hit "PAUSE EVERYTHING" — recoverable, no cycle running | operator action |

`error` and `paused` are distinct: `error` is system-detected and tied to an env condition; `paused` is operator-initiated. Resume flow differs (see §3 "Pickup vs. resume").

**`cycles.stage`** (which stage of the loop a single attempt is in):

| Value | Meaning |
|---|---|
| `planning` | Currently calling the model to produce plan + diff |
| `plan_review` | Plan generated, parked waiting for human approval (only if policy requires) |
| `building` | Diff being applied + lint/type/build/test gates running |
| `pr_review` | PR opened, parked waiting for human merge (only if policy requires) |
| `done` | Cycle reached terminal success — `cycles.outcome='merged'` |
| `cancelled` | Cycle reached terminal failure — see `cycles.outcome` for which kind |

**`cycles.outcome`** (terminal disposition of a single cycle):

| Value | Meaning |
|---|---|
| `merged` | PR merged. Counts toward Receipts. |
| `vetoed` | Human rejected at a review gate. |
| `failed` | Soft failure that consumed an iteration but did not produce a revise (e.g. budget exhausted on this cycle's failure). |
| `error` | Hard failure. The work item is set to `status='error'`. The cycle row is closed; on resume, a new cycle is created (iteration unchanged). |
| `revised` | Cycle ended because a revise was triggered; a new cycle row exists with `iteration+1`. |
| NULL | Cycle is still in progress. |

### Input adapters

All three push into `work_items` and never bypass it.

| Adapter | Trigger | Notes |
|---|---|---|
| **UI / API** (primary) | `POST /api/admin/architect/work-items` | New "Architect → Queue" page. Form fields = schema fields. Local-first ethos. |
| **Markdown** (`.architect/queue.md`) | File watcher in `coastal-daemon`, debounced 2s | Format: H2 = title, body = body, fenced YAML for `target_hints / acceptance / budget / approval_policy / review_timeout_min / on_timeout`. Idempotent — uses content hash as `source_ref`. |
| **SKILL.md** (`.architect/skills/*.md`) | Same file watcher | Portable [SKILL.md](https://github.com/agent0ai/agent-zero) format compatible with Agent Zero / Space Agent ecosystem. Frontmatter declares the work item; body holds the plan body. Adopting the standard makes Coastal.AI skills/work-items interchangeable with the broader Agent Zero ecosystem (35k+ combined stars). One file = one work item. See "SKILL.md format" subsection below for schema. |
| **GitHub Issues** (opt-in) | Cron poll every 5 min if `GITHUB_TOKEN` set | Pulls issues with label `architect-ready` from configured repo. Closes loop by commenting the PR URL back on the issue and adding label `architect-pr-open`. |
| **Curriculum** (internal) | When queue is empty AND mode ∈ {hands-off, autopilot} AND `curriculum.enabled = true` | Architect generates its own work items from codebase analysis at low priority. See §3 "Curriculum mode" for the scan procedure. Always inserts with `approval_policy='full'` regardless of global Mode — items the architect requested of itself get extra rigor. |

### SKILL.md format (Agent Zero / Space Agent compatible)

Adoption of the open SKILL.md standard from the Agent Zero / Space Agent ecosystem. One `.md` file = one work item, parseable by other tools that speak the same format.

```markdown
---
name: Add retry to web tool
description: Wrap http_get in exponential-backoff retry on 5xx responses
version: 1
locale: en
target_hints:
  - packages/core/src/tools/core/web.ts
acceptance: |
  - http_get retries up to 3 times on HTTP 5xx
  - Backoff: 100ms, 400ms, 1.6s
  - On final failure, error includes attempt count
budget_loc: 80
budget_iters: 3
approval_policy: plan-only
---

# Why

A common cause of agent task failures is transient 5xx from external APIs.
Adding a small retry layer at the http_get boundary should fix most of them
without changing call sites.

# Hints

The existing pattern in `packages/core/src/tools/core/sqlite.ts` for
SQLITE_BUSY retry is a good reference — same shape, different errors.
```

**Parser rules:**

- Frontmatter is YAML between `---` markers. Required: `name`, `description`. All other fields fall back to schema defaults.
- Body (`# Why`, `# Hints`, free prose) becomes `work_items.body`.
- One file per work item. Filename is `source_ref` for idempotency.
- File watcher uses content hash for change detection. Edits to a file with an in-progress cycle are queued for the *next* iteration's revise context, not applied retroactively.

**Skills library export/import** (separate from work items, but same format):

The existing Skills feature (v1.2.0) gets a SKILL.md export/import path so that prompt-template skills become portable. New routes:

```
GET  /api/admin/skills/:id/export        → SKILL.md text
POST /api/admin/skills/import            { content: string } → upsert
GET  /api/admin/skills/export-pack       → tar.gz of all skills as SKILL.md files
```

This is the foundation for the v1.6 "skill marketplace" / Agent Zero plugin index integration. For v1.5 we ship export/import only — discovery is manual (drop a file, click import).

### State machine

```
                              ┌──────── revise (with structured failure ctx) ────────┐
                              │                                                       │
                              ▼                                                       │
pending  ──▶  planning  ─────▶  plan_review  ──▶  building  ──▶  pr_review  ──▶  merged
                  │                  │                 │              │
                  │ soft fail        │ human reject    │ soft fail    │ human reject
                  │ (parse / diff /  │  or revise      │ (apply /     │  or revise
                  │  budget LOC /    │                 │  test / lint │
                  │  locked path)    │                 │  / type /    │
                  ▼                  ▼                 │  build)      ▼
              [budget?]          [reject→cancel]       ▼          [reject→cancel]
                  │              [revise→loop]    [budget?]       [revise→loop]
                  │                                    │
                  ├── yes → revise (loop back to planning, iteration++)
                  │
                  └── exhausted → cancelled

[hard fail at any stage] ──▶  error (paused, no iteration consumed)
                                │
                                └── human fixes env → resume
```

### Failure taxonomy (drives revise vs. pause)

| Failure | Type | Behavior |
|---|---|---|
| Planner returns no/invalid diff block | soft | revise |
| Diff doesn't apply (conflict) | soft | revise |
| Locked-path violation | soft | revise |
| `budget_loc` exceeded | soft | revise (smaller change) |
| Tests fail | soft | revise |
| Typecheck fails | soft | revise |
| Lint fails | soft | revise |
| Build fails | soft | revise |
| Network error to LLM (after retry) | hard | pause → `error` |
| `gh` CLI missing or auth failed | hard | pause → `error` |
| Filesystem permission denied | hard | pause → `error` |
| SQLite write failure | hard | pause → `error` |
| Branch creation race (after 1 retry) | hard | pause → `error` |

### Revise mechanics

- Three revise edges, all bounded by `work_items.budget_iters` (default 5):

| Edge | Trigger | New cycle context |
|---|---|---|
| `plan_review → planning` | Reviewer clicks "Revise" with comment | Prior plan + reviewer comment |
| `building → planning` | Soft build/test failure | Prior plan + diff + truncated failure output |
| `pr_review → planning` | Reviewer requests changes | Prior plan + diff + PR review comments |

- Each revise spawns a new `cycles` row, linked by `work_item_id`, with `iteration` incremented.
- A `reject` decision *always* terminates regardless of remaining budget.
- Cooldown between iterations: `min(2^iteration s, 60s)` to prevent thrash.
- Comment is mandatory for human-triggered revise.

### Package boundaries

- `packages/architect/src/` — major rewrite. New files: `queue.ts`, `stage-runner.ts`, `model-router-client.ts`, `pr-creator.ts`, `adapters/{ui,markdown,github}.ts`. Existing files preserved and extended: `planner.ts`, `patcher.ts`, `validator.ts`, `announcer.ts`, `config.ts`, `index.ts`.
- `packages/core/src/architect/` — store + types **owned by core**. The `packages/architect` package depends on `packages/core` (already true in the workspace) and imports types and store APIs from here. **Core does not import from `packages/architect`** — this is the rule that prevents a circular dependency. Concretely:
  - `packages/core/src/architect/types.ts` — pure type definitions (work item, cycle, approval, status enums). Imported by both core HTTP routes and the architect daemon.
  - `packages/core/src/architect/store.ts` — SQLite read/write functions. Imported by both.
  - `packages/core/src/architect/metrics.ts` — `cycle_metrics` rebuild logic. Triggered by core (HTTP) on cycle terminal events; the architect daemon emits the events but doesn't recompute.
  - `packages/architect/src/*` — orchestration only. Imports types/store/metrics from core, never the other way.
- `packages/core/src/api/routes/architect.ts` — admin routes (queue, cycles, insights, receipts, callbacks).
- `packages/core/src/channels/architect-notify.ts` — formats and sends gate / digest notifications via existing `ChannelManager`.
- `packages/web/src/pages/Architect/` — new pages: `Wizard.tsx`, `StatusCard.tsx`, `Queue.tsx`, `Cycle.tsx`, `Activity.tsx`, `Insights.tsx`, `Receipts.tsx`, `Settings.tsx`.
- `packages/architect/bin/coastal-ai-architect.js` — CLI entrypoint, called by all three OS wrappers.
- `coastal-ai.cmd`, `coastal-ai.ps1`, `coastal-ai` — extended to forward `architect <subcommand>` to the bin.

## 3. Loop Mechanics

### Pickup

The architect daemon polls `work_items` every 60 s for `status='pending'` items, ordered by `created_at ASC`. Single-cycle invariant — no concurrent architect runs against the same repo (lockfile `data/.architect-pid` already exists).

### Pickup vs. resume (hard-fail recovery)

Items in `status='error'` or `status='paused'` are **not** picked up automatically. Resume requires an explicit signal — restart alone is not sufficient (a daemon restart with the env still broken would just re-fail).

Two resume signals, both set `work_items.resumable=1` and clear `paused_reason`:

1. **UI / CLI resume** — operator clicks "Resume" in the Architect → Cycle page or runs `coastal-ai architect resume <id>`. Optional preflight: the runner re-checks the original failure condition (e.g. re-ping `gh auth status`, re-test `data/architect.db` write) and refuses to resume if the env is still broken, returning the same hard-fail reason.
2. **Bulk resume after pause** — operator clicks "Resume all" in the Power switch dropdown after a global PAUSE. Same preflight per item.

**Cycle bookkeeping for hard fails:**

- The cycle that hit the hard fail closes with `outcome='error'`, `failure_kind='env_*'`, `error_message` populated. `cycles.iteration` is **not** incremented for hard fails — the iteration counter only advances on soft revises.
- On resume, a **new** cycle row is created with the same `iteration` number as the failed one (so `budget_iters` is unaffected). `cycles.revise_context` for the new cycle is `{ resumed_from: <prior_cycle_id>, reason: 'env_resume' }`.
- If the same hard fail recurs three times across resumes, the work item escalates: `paused_reason` updates with the recurrence count and a new field `escalated_at`. The UI surfaces this as "this keeps failing — please investigate."

**No restart-as-resume.** A daemon restart on a host with an `error`-status work item leaves it in `error`. The resume path is always explicit.

### Stages

#### `planning`

Inputs: work item + optional `revise_context` + repo source snippets.

- **Source selection:** if `target_hints[]` is set, read those files. Else, the planner uses `grep` over the repo for keywords from `title + body` to assemble candidate files. Total context capped at 12 k tokens.
- **Single LLM call:** prompt asks for `<plan>...</plan><diff>...</diff>` blocks. Plan is prose, diff is unified format (multi-file allowed).
- **Validation:** diff parses, all touched files exist (or are creates), no locked-path hits, `budget_loc` not exceeded.
- **Failure modes:** parse error, locked-path, budget overflow → soft revise.

#### `plan_review` (gated by policy)

- Skipped if `approval_policy ∈ {pr-only, none}`.
- For `plan-only` (default) and `full`: SSE event broadcasts to the web UI. Cycle parks until reviewer hits Approve / Revise / Reject in the **Architect → Cycle** page or via Output-Channel callback.
- Timeout: configurable per-item (`review_timeout_min`, default 60 min). On timeout: behavior from `on_timeout` field. Default `reject` for safety.

#### `building`

Sequenced gates, each a soft-fail candidate with its own `failure_kind`. Lint / typecheck / build / test run **only on packages touched by the diff** (deduce from changed files → workspace map).

**Time-Travel snapshot capture** (taken automatically at the start of every `building` stage, before `git apply`):

The architect maintains a *shadow* Git repository at `data/.architect-snapshots/` that mirrors the workspace state. It is **separate from** the real `.git` — never confused with normal commits, never visible to `git log` on the project. Captures are:

| Trigger | When | `captured_by` | Default retention |
|---|---|---|---|
| `auto:plan` | After plan accepted, before any file write | runner | short (30d) |
| `auto:building` | At start of `building` stage, before `git apply` | runner | short (30d) |
| `manual` | Operator clicks "Snapshot now" in UI | operator | long (1y) |

Each snapshot is a `git commit-tree` of the entire workspace tree, parented to the previous snapshot. The shadow repo never modifies the actual `.git` working tree — it operates via `GIT_DIR=data/.architect-snapshots/.git GIT_WORK_TREE=<repo-root>` invocations.

**Restore mechanism (operator-driven):**

From the Cycle page or Activity page, operator selects a snapshot and clicks "Restore":

1. Runner creates a new branch in the *real* repo: `architect/restore-<snapshot_id>`.
2. Runner checks out the snapshot's tree onto that branch with a single commit: `revert(architect): restore workspace to <captured_at>`.
3. Operator is shown a diff view (workspace now vs. snapshot) and a notice: *"This is on a recovery branch. Merge to main yourself if you want to keep it."*
4. **Restore never auto-merges to main** — even on `Autopilot` mode. Restoration is a deliberate, human-confirmed action.

**Retention:**

- `short` (default): snapshots older than 30 days pruned at daemon start.
- `long`: kept for 1 year.
- `pinned`: never pruned. Operator pins via UI; pinning a snapshot also pins all its ancestors (so pin-then-prune doesn't break the chain).

**Storage cost:** Each snapshot adds ~1 commit object + tree delta — typically < 50KB for a code-only workspace. 100 snapshots ≈ 5MB. Disk pressure is bounded by the 30-day default retention.



| Gate | Command | failure_kind |
|---|---|---|
| Branch create | `git checkout -b feature/architect/<work-item-id>-<slug>` | env_branch |
| Apply diff | `git apply` (existing `Patcher`) | apply |
| Lint | `pnpm --filter <touched-pkg> lint` | lint |
| Typecheck | `pnpm --filter <touched-pkg> exec tsc --noEmit` | type |
| Build | `pnpm --filter <touched-pkg> build` | build |
| Tests | `pnpm --filter <touched-pkg> test` | test |

Stops at first failure. Output captured (truncated to 4 k chars) into `revise_context`.

#### PR creation (post-build, pre-`pr_review`)

- Commit:
  ```
  chore(architect): {work_item.title}

  Work item: {id}
  Iteration: {n}

  Co-Authored-By: coastal-architect <bot@coastal.ai>
  Architect-Cycle: {cycle-id}
  ```
- Push: `git push -u origin <branch>`.
- PR open: `gh pr create --title "[architect] <title>" --body <generated-body> [--draft if approval_policy=full]`.
- PR body: linked work item, plan text, test summary, model used, iteration count, attribution.
- For GH-issue source items: comment PR URL back on issue + add label `architect-pr-open`.
- For markdown source items: replace the work-item entry in `.architect/queue.md` with `<!-- moved to PR #N -->`.
- `gh` missing or auth failed → hard error (paused, not revised — not the LLM's fault).

**Hard-fail recovery for PR creation specifically.** The build was green, so the diff is committed on a branch and pushed (or pushed-pending). On resume after fixing `gh`:
1. Runner detects the cycle was last in `building` stage with `outcome='error'`, `failure_kind='env_gh'`.
2. Branch state is checked: if branch exists locally and on remote, runner skips re-build and re-tries `gh pr create` directly.
3. If branch was pushed but commit got force-overwritten by an unrelated process (rare), the runner re-runs the full `building` stage on a new branch. This is the **one explicit exception** to the "hard-fail resumes don't consume iterations" rule: because the diff itself must be regenerated (the original work is gone), it counts as a soft revise consuming an iteration, with `revise_context.reason='env_recovery_branch_lost'`. Documented here so the failure-taxonomy table's "hard fail = no iteration consumed" rule remains the default and this is a named exception.
4. If `gh pr create` succeeds on resume, cycle proceeds to `pr_review` as if nothing happened.

The same recovery pattern applies to push failures (network issue mid-push): if `git push` fails, `failure_kind='env_push'`, resume re-attempts the push.

#### `pr_review` (gated by policy)

- `none`: architect auto-merges as soon as CI is green via `gh pr merge --squash --auto`.
- `pr-only` / `plan-only`: PR parks; merge happens when human merges through GitHub or clicks "merge" in the dashboard. Dashboard polls PR status every 30 s.
- `full`: PR opens as **draft**, requires human ready-for-review flip first.

### Model routing

Routes through the existing `ModelRouter`. New routing domain `architect` registered, three priority levels mapped to project tiers:

| Stage call | Domain | Priority | Tier preference |
|---|---|---|---|
| Plan + diff generation | `architect` | `high` | vLLM (if up) → AirLLM → Ollama largest available |
| Commit-message / PR-body summarisation | `architect` | `low` | Ollama smallest tagged `summary-ok` |
| Failure classification (when ambiguous) | `architect` | `low` | same |

Falls back to Ollama default if no tiered models registered. Model used per cycle is recorded in `cycles.model_used`.

**Minimum-capability gate (startup check).** When the architect daemon starts, it queries `ModelRouter` for the model that would handle `architect:high`. If that model's tier is below the configurable minimum (`CC_ARCHITECT_MIN_TIER`, default `medium`), the daemon **refuses to start** and writes a clear message to logs and to the dashboard's "What's happening right now" card:

> *"Architect can't run on this hardware tier. The smallest model available for high-priority planning is too small to produce reliable diffs. Install or download a stronger model, or set `CC_ARCHITECT_MIN_TIER=low` to run anyway (not recommended)."*

The tier mapping uses the existing `Hardware-Aware Tiers` system already documented in the README (Lite / Standard / Apex). Without this gate, sub-capable models cause every cycle to revise-until-budget-exhausted, masking the real problem.

### Locked-path policy (expanded)

Default deny list (regex / glob, configurable in `data/.architect-locked-paths`):

```
data/**
.git/**
node_modules/**
.env*
**/secrets/**
packaging/**
coastalos/build/**

```

Override per work item via `allow_self_modify: bool` (admin-created work items only).

### Curriculum mode (idle-time work-item generation)

Inspired by the Curriculum/Executor symbiotic-evolution idea from the Agent0 paper (UNC / Salesforce / Stanford, 2025), adapted to Coastal.AI's inference-only constraints — no model training, just well-grounded prompting against real codebase state.

**When it runs:**

- Architect daemon is `ON`.
- `work_items` queue is empty (no `pending`, `active`, or `awaiting_human` items).
- Mode is `hands-off` or `autopilot` (not `hands-on` — too noisy for that mode).
- `curriculum.enabled = true` AND today's curriculum budget is not exhausted.

If all conditions hold, a **scan cycle** runs once per `curriculum.scan_interval_min` (default 4 hours).

**Scan cycle (different from a normal cycle — never modifies code, only writes work items):**

1. **Telemetry harvest:** read recent `cycle_metrics`, `action_log` failures, `pnpm test` reports, `tsc` warnings, lint output, dead-code reports.
2. **Codebase signals (cheap heuristics, no full LLM scan):**
   - `git log --since=30d` → identify churn-heavy areas (more bug-prone, more candidates for cleanup)
   - `grep -nE "TODO|FIXME|XXX" --before-30-days` → stale TODOs
   - Untouched files in low-coverage packages
3. **One LLM call** (`architect:high` priority): given the harvested signals as context (capped at 8k tokens), the model proposes 0-N candidate work items as SKILL.md frontmatter. Prompt explicitly says: *"only propose changes you have specific evidence for in the inputs above. Reject anything you'd be guessing."*
4. **Validation:** each proposal must reference a real file (verify via `fs.statSync`), respect locked-paths, and have plausible `budget_loc < 200`.
5. **Insertion:** validated proposals inserted into `work_items` with `source='curriculum'`, `approval_policy='full'` (always plan-review at minimum, regardless of global Mode), `priority` field set to `low`.
6. **Idempotency:** before inserting, check no existing pending/active item has the same `target_hints` overlap + similar title (cosine similarity on title alone, threshold 0.85). Prevents the architect from proposing the same fix repeatedly.

**Three intensity levels** (Settings → Architect → Curriculum):

| Level | `curriculum.enabled` | Daily budget | What it'll propose |
|---|---|---|---|
| `off` (default) | false | 0 | Curriculum doesn't run. |
| `conservative` | true | 1 item / day | Only obvious wins: stale TODOs older than 30d, untested public functions in low-coverage packages, lint warnings repeated > 3 times. No refactors. |
| `aggressive` | true | 5 items / day | Above plus: small refactor proposals when a function exceeds 80 LOC, dependency-update opportunities flagged in `pnpm audit`, dead-code candidates from existing dead-code detector. Still no architectural changes. |

**Safety rails (curriculum-specific):**

- All curriculum-sourced items are flagged in the UI with a distinctive `🌱 curriculum` badge so the operator always knows whether the architect proposed this to itself.
- Curriculum items can never modify `packages/architect/**` (no `allow_self_modify` override possible — admin-only flag is unreachable from the curriculum path).
- A failed curriculum item (`outcome='vetoed'` or `'failed'`) suppresses similar proposals for 30 days (matched by target_hints + title similarity).
- Insights page tracks **curriculum acceptance rate**: `curriculum-merged / curriculum-proposed`. Below 25% → toast warning to operator suggesting `aggressive` → `conservative` downgrade.

**Schema additions:**

```sql
ALTER TABLE work_items ADD COLUMN priority TEXT NOT NULL DEFAULT 'normal';  -- 'low' | 'normal' | 'high'

CREATE TABLE curriculum_suppressions (
  id              TEXT PRIMARY KEY,
  signature       TEXT NOT NULL,         -- normalized title + sorted target_hints
  suppressed_until INTEGER NOT NULL,
  reason          TEXT NOT NULL,         -- 'vetoed' | 'failed' | 'duplicate'
  created_at      INTEGER NOT NULL
);
```

**Concurrency:** curriculum scans share the same `data/.architect-pid` lock as normal cycles. Only one runner runs at a time. A scan that finds 0 valid proposals (validation failed everything) emits a no-op `cycles` row with `stage='curriculum'`, `outcome='failed'` for observability — Insights page can graph "scan attempts vs. items produced."

### Concurrency safety

Single `data/.architect-pid` lockfile (already exists). Architect refuses to start if file present and PID alive. The HTTP route `/api/admin/architect/run` also acquires this lock.

**Manual cycle triggers** — three ways to start a cycle outside the 60s daemon poll, all serialized through the same lockfile:

| Trigger | Platform | Mapped to |
|---|---|---|
| `kill -USR1 <pid>` | Linux / macOS | Existing legacy entry-point retained for the dogfood. New architect also honors USR1 → "process next pending work item." |
| Settings → "Run Now" button | All platforms | `POST /api/admin/architect/run` — preferred path. |
| `coastal-ai architect tick` (CLI hidden subcommand) | All platforms | Wraps the same HTTP route. |

Windows never depends on USR1 — the HTTP route is the only mechanism. Dogfood Phase 3 instructions reflect this: USR1 on Linux/macOS, "Run Now" button on Windows.

## 4. Operator UX (Non-Technical)

### UI design principles (Space Agent-inspired)

The new Architect pages set the visual / interaction baseline for v1.5.0 — and become the reference template for retrofitting older pages in v1.6+. Borrowing the parts of Space Agent's flow that work without copying their architecture:

| Principle | What it means concretely | Where it shows |
|---|---|---|
| **Inline expansion over navigation** | Clicking a row expands it in place. No "details page" routes for things that fit in a card. Activity rows expand to show plan + diff + tests. | Activity, Cycle, Queue |
| **Canvas, not pages** | Long-running views feel like a scrollable canvas with floating cards, not a stack of paginated lists. Pagination only when content genuinely overflows. | Activity, Receipts |
| **One-line answers up top** | Every page opens with a single sentence stating the current state. Details follow only if the user wants them. | Status card, Insights tiles |
| **State preserved per user** | Expanded vs. collapsed state, filter selections, "show resolved" toggles persist across navigation per logged-in user. Nothing resets when you click around. | All pages, stored per-user in `user_preferences` table (existing) |
| **Progressive disclosure** | Power-user controls (model used, locked-paths config, raw cycle JSON) live behind a single "Show details" disclosure, never default-visible. | Cycle page, Settings |
| **Visible motion only when meaningful** | Pulse on the active cycle dot, fade-in on new activity rows, no other gratuitous animation. Reduced-motion respected. | Activity timeline, Status card |
| **Mobile-respecting** | Approve / Revise / Reject buttons readable and tappable on a 375px-wide phone (the Telegram → dashboard deep-link path). | Approval gate UI, Cycle page |
| **Empty states are doorways** | Empty queue → big illustrated card with "Add a task" CTA, not just "No items." Empty Receipts → "The architect hasn't shipped yet — here's how to get its first PR." | Queue, Receipts, Activity |

**Non-goals for v1.5:**

- Not adopting Space Agent's "agent reshapes the UI" runtime — that's v2.x territory if at all. The UI is human-built.
- Not retrofitting existing Coastal.AI pages (Chat, Dashboard, Pipeline, etc.) in v1.5 — only the new Architect pages get this treatment. Retrofit wave is v1.6 candidate.

**Style tokens:** the existing DESIGN.md tokens (Deep Ocean Command Center, electric cyan, glassmorphic surfaces) carry over unchanged. The principles above are about *information architecture*, not look-and-feel — they're meant to coexist with the existing aesthetic.

### Secrets & token storage (resolved)

Two secrets are introduced by this design. Both live under `data/` (the project's existing data dir, already gitignored, encrypted on supported platforms when running with `CC_DATA_DIR` on encrypted storage per existing security guidance).

| Secret | File | Purpose | Rotation |
|---|---|---|---|
| Architect callback signing key | `data/.architect-callback-key` (32 random bytes, written 0600) | HMAC-signs Output-Channel approval callback URLs | Auto-rotated on daemon start if file is older than 30 days; in-flight callbacks invalidated. |
| GitHub adapter token | `data/.gh-token` (read-only by daemon user) | Auth for GitHub Issues poll + `gh` CLI fallback | Manual; UI surfaces "expires soon" warning if token has `expires_at` field set. |

The signing key is generated once on first daemon start (`crypto.randomBytes(32)`) — never user-supplied. The GH token is user-supplied via the Settings page and never echoed to logs.

**File permission enforcement (cross-platform):**

| Platform | Mechanism | Enforcement on daemon start |
|---|---|---|
| Linux / macOS | POSIX `0600` — owner read/write only | `fs.statSync(file).mode & 0o077` must be `0`; otherwise daemon refuses to start with explicit message. |
| Windows | NTFS ACL — only the daemon-running user (and SYSTEM/Administrators) granted read | Use `icacls` (or Node's `node:fs/promises` + `winapi-bindings` if added) to set/verify ACL: explicit DENY for `Everyone` and `Users`, ALLOW for current user only. Verification on start: enumerate ACLs, refuse if any non-owner principal has read. |

If the platform check fails, the daemon emits a remediation snippet (`chmod 600 …` or `icacls … /inheritance:r /grant:r "%USERNAME%":F`) instead of just refusing silently.

Callback URLs are short-lived: the HMAC payload includes `cycle_id`, `gate`, and `expires_at` (TTL = `work_items.review_timeout_min`). After expiry, callback hits return `410 Gone` with a deep-link to the dashboard.

### Where data comes from

No new write paths beyond §2 schema. All three pages are read-only views over `work_items`, `cycles`, `approvals`, `cycle_metrics`, plus `git log` (filtered by author = `coastal-architect <bot@coastal.ai>` or commit-trailer presence).

**`cycle_metrics` rebuild triggers** (no new scheduler — uses existing infrastructure):

1. **On every cycle terminal transition** (`done`, `cancelled`, `error`) — runner emits a debounced (5 s) `metrics:rebuild` event. The metrics rebuilder consumes the event, recomputes only the affected `date` row(s), and writes back. Cheap and incremental.
2. **On daemon start** — full rebuild from scratch if the metrics table is empty or `MAX(date)` older than 7 days (catches drift from manual DB edits / restores).
3. **No nightly cron.** The project's existing `coastal-daemon` runs `croner` for *user* jobs. We deliberately avoid adding a system cron there — the on-write triggers are sufficient and have no scheduler dependency.

**Receipts denominator math** (defined explicitly):

The Receipts page percentages use `git rev-list <prior-tag>..HEAD` as the universe, where `<prior-tag>` is the tag matching the `release` query parameter's predecessor (e.g. `v1.5.0` → `v1.4.0..HEAD`). Excluded from the denominator: merge commits (`--no-merges`), commits whose tree-only change is `CHANGELOG.md` / `VERSION` (release admin commits), and any commit on a branch not yet merged to the release tag's branch. Architect-attributed commits are identified by `Co-Authored-By: coastal-architect <bot@coastal.ai>` trailer presence (counted), and LOC contribution by `git diff --shortstat` of those commits' patches against their parents.

## 6. Reliability Dogfood Plan

### Sequencing

Old architect runs first, in parallel with new architect being built — no bootstrap dependency. The two streams meet in the v1.5.0 release branch.

### Phase 1 — Foundation

> **Note:** Phase 1 changes are **human-built**, landed before any architect run. They modify `packages/architect/**` which is on the new architect's locked-paths list (§3) — that's not a contradiction; the locked-paths list governs what the *new* architect may auto-modify, and Phase 1 is preparing the runway for both architects. The `allow_self_modify` work-item flag exists for the case where a human wants to use the new architect to modify its own package later (admin-only, opt-in per item).

Two tiny pre-changes to the *current* architect, before any v1.5.0 work begins:

1. **Trailer-aware commits.** `packages/architect/src/patcher.ts → commitChange()` adds:
   ```
   Co-Authored-By: coastal-architect <bot@coastal.ai>
   Architect-Cycle: <cycle-id>
   ```
   The Receipts page in §5 parses these.
2. **`scripts/seed-skill-gaps.ts`.** New one-shot Node script:
   - Reads `data/action_log` (existing SQLite table written by `AnalyticsStore`).
   - Selects rows where `success=false AND tool_name IN ({run_command, read_file, write_file, query_db, http_get, git_status}) AND timestamp >= $1`.
   - Groups by tool + a normalized error pattern (first line of `error_message` with numbers / paths / IDs masked).
   - Inserts deduped rows into `data/skill-gaps.db`.
   - Also accepts `--seed-curated` to load handcrafted gaps from `scripts/seed-skill-gaps.curated.json`.

### Phase 2 — Curated seed file

`scripts/seed-skill-gaps.curated.json`:

```json
[
  {
    "tool_name": "run_command",
    "failure_pattern": "long-running command exceeds default 30s timeout with no signal propagation"
  },
  {
    "tool_name": "read_file",
    "failure_pattern": "path traversal via .. segments not blocked at agent workspace boundary"
  },
  {
    "tool_name": "query_db",
    "failure_pattern": "concurrent write returns SQLITE_BUSY without retry-on-busy"
  },

    "tool_name": "http_get",

  },

  }

```

### Phase 3 — Dogfood run

Branch: `feat/architect-dogfood-v1.5` (fanned off `master` at the start of v1.5.0 work).

Operator runs, in order:

```bash
# 0. Ensure trailer-aware patch and seed script have landed
git checkout -b feat/architect-dogfood-v1.5

# 1. Telemetry replay — pulls real failures from your action_log
node scripts/seed-skill-gaps.ts --since "2026-01-01"

# 2. Curated seeds — adds handcrafted gaps
node scripts/seed-skill-gaps.ts --seed-curated

# 3. Engage autonomous mode (bypass veto window for the dogfood run)
touch data/.architect-autonomous

# 4. Run the architect with manual triggers (one cycle per gap)
kill -USR1 $(cat data/.architect-pid)        # Linux/Mac
# Windows: open the Architect dashboard and click "Run Now" 5 times

# 5. Inspect commits on the branch
git log --oneline --author="coastal-architect"
```

The architect uses the existing veto-window mechanism (`.architect-autonomous` flag bypasses human prompt). All gates remain in effect: must produce parseable diff, diff must apply cleanly, tests must pass. Failures don't merge.

> **Legacy-only flag.** `data/.architect-autonomous` is honored **only by the legacy `skill-gaps`-driven cycle path**. The new queue-driven architect (the v1.5.0 main feature) ignores this file entirely — it has its own per-work-item `approval_policy` and the global Mode switch. We document this here to prevent confusion when both daemons run on the same branch during the dogfood phase. Post v1.5.0, the legacy path is retained behind `--legacy` for one more release before deprecation.

### Phase 4 — Acceptance review (human)

For each commit on `feat/architect-dogfood-v1.5` authored by `coastal-architect`:

| Check | Pass criteria |
|---|---|
| Tests green | Re-run on merge target (`master @ HEAD`) to catch drift |
| Diff sane | No hallucinated APIs, no removed tests, no `// TODO` left in place of logic, no comment that contradicts the code |
| Real fix | Cross-check against gap row — does the commit actually address the seeded failure pattern? |
| LOC | < 80 LOC changed (the loop's effective ceiling for quality) |

Anything failing those gets dropped. Surviving commits cherry-picked into v1.5.0 release branch with `(cherry picked from <sha>)` trailer added on top of existing trailers.

## Risk Register

| Risk | Likelihood | Mitigation |
|---|---|---|
| Old architect produces 0 mergeable commits | Medium | Success bar table written up front, CHANGELOG language committed for that case. |
| Multi-file diff support harder than expected (planner output reliability) | Medium | Start with single-file work items, multi-file is incremental — gate with a feature flag. |
| Three input adapters (UI + MD + GH) is too much surface for one release | Medium | Markdown + GH adapters are pluggable; ship UI-only if either runs late. |
| `gh` CLI dependency adds friction for users who don't have it | Low | First-run wizard detects and offers Win/Mac/Linux install commands. |
| Telegram approve-from-phone signed-callback URL exploited | Low | Use HMAC-signed tokens with short TTL (60 min — same as default `review_timeout_min`); single-use. |
| Self-modify guard accidentally blocks legitimate architect-on-architect work | Low | `allow_self_modify` override on work item; admin-only. |
| Daily digest channel spam | Low | Off by default; opt-in in wizard. |
| Cycle metrics aggregation lags merge | Low | Rebuild triggers on every cycle terminal transition (debounced 5s) plus on daemon start. No scheduler. |
| Operator hits "Autopilot" by accident | Low | Confirmation dialog with explicit risk text; logged with user id. |
| Locked-paths config drifts from project structure over time | Low | Default list lives in code (`config.ts`); user override in `data/`; checked at daemon start. |
| Curriculum mode proposes the same fix repeatedly (loop on rejection) | Medium | `curriculum_suppressions` table holds 30-day bans on (title-similarity + target_hints) signatures after vetoed/failed cycles. |
| Curriculum mode proposes hallucinated improvements not grounded in the codebase | Medium | Each proposal must reference a real file (`fs.statSync` validation pre-insert); LLM prompt explicitly tells the model to reject anything it'd be guessing about; acceptance rate tracked, < 25% triggers downgrade-to-conservative toast. |
| SKILL.md spec drifts between Coastal.AI and Agent Zero | Low | Adopt the format strictly via parser; if Agent Zero / Space Agent diverge, treat that as a v1.6 reconciliation task (same backlog as #5). Frontmatter is forward-compatible (unknown keys ignored, not rejected). |
| Time-Travel snapshot disk usage grows unbounded | Low | Default 30-day retention with daemon-start prune; `pinned` snapshots retain ancestors; storage bounded ~5MB per 100 snapshots in code-only repos. UI surface for "you have N pinned snapshots, oldest from <date>" warning at > 50 pinned. |
| Time-Travel restore path confused with normal git revert | Medium | Restore *always* creates `architect/restore-<id>` branch — never touches main directly. Operator sees a deliberate "Recovery branch created. Merge yourself if you want." message. Architect 2.0 cannot auto-merge restore branches even on Autopilot. |
| Adopting Space Agent UI principles confuses returning users | Low | Principles only apply to *new* Architect pages — Chat / Dashboard / Pipeline etc. unchanged in v1.5. Retrofit wave is its own v1.6 release with its own changelog/migration notes. |
